General Information

What is a 'Hash Algorithm'
A hash function is simply an algorithm that takes a string of any length and reduces it to a unique fixed length string. In other words, they turn any amount of data into a fixed-length "fingerprint" that cannot be reversed.

What is a cryptographic hash function?
The ideal hash function has three main properties:

  • Consistent: hash(X) always yield same result
  • One-Way: Given Y, hard to find X such that hash(X) = Y
  • Collision Resistant: It is extremely difficult to find a collision.
Collision is further explained in detail here!

Here is a quick example:

'test' => SHA-1 => 'a94a8fe5ccb19ba61c4c0873d391e987982fbbd3'

Consistent: Every time that anyone anywhere runs the word 'test' through the SHA-1 function, they should always get: a94a8fe5ccb19ba61c4c0873d391e987982fbbd3. This is not as secure as it might for reasons desribed here.

One-Way: Also, it should be computationally infeasible to find any other word which also hashes to a94a8fe5ccb19ba61c4c0873d391e987982fbbd3.

Collision-Resistant: Finally, if anyone were to give you only 'a94a8fe5ccb19ba61c4c0873d391e987982fbbd3' and tell you that it came from the SHA-1, you should have absolutely no way to figure out what was put into the function to create that.

Where are Hashes used?

  1. Cryptographic Signature
    A cryptographic signature, also called a digital signature is a key component of establishing trust of a digital identity. A digital signature essentially breaks down to a encrypted hash value (first, a set of contents are hashed and second, the hash digest is encrypted). One of the most popular technolgies where digital signatures are used is Public Key Infrasturce. The trust aspect of digital signatures arises from the fact that decryption of the signature is only possible by the public key whose corresponding private key was used to create the signature (decrypt the singature). And if the owner of the key pair is someone you trust (Certificate Authority), you are translating the trust to the digital signature as well. Hashing serves the same purpose here which is to maintain the integrity of the data (in this case the contents of the certificate). Read more about digital signatures here:

  2. Commitment Schemes
    A commitment scheme is a method to prove the ownership of a particular message to its true owner. For example, imagine a game of rock, paper and scissors being played online. We need to make sure that player 1 and player 2 see each other's choice without causing any unfair advantage (for example, if the game was being played through email, player 2 will see what player 1 chooses and can respond back accordingly), commitment schemes allow a way around this. Learn more

  3. Generate Message Authentication Codes
    The message authentication code (MAC) is generated from an associated message as a method for assuring the integrity of the message and the authenticity of the source of the message. A secret key to the generation algorithm must be established between the originator of the message and its intended receiver(s). There are currently three MAC generation algorithms recognized by NIST and they are: HMAC, KMAC and CMAC. Read more about the algorithms here.

  4. Key Derivation Functions
    A Key derivation function (KDF) is a basic and essential component of cryptographic systems: Its goal is to take a source of initial keying material, usually containing some good amount of randomness, but not distributed uniformly or for which an attacker has some partial knowledge, and derive from it one or more cryptographically strong secret keys.

  5. Store Passwords Digest
    Instead of storing plain text password in the database one way is to store the hash of the password. You might be thinking that if we cannot get the actual password back from the hash then how are we going to verify the credentials that the user entered? It’s simple, apply the same hash function on the password which user entered and then compare it with the hash stored in the database. If both hashes match then the user is authenticated (since hash of same input will give same output). Now if the attacker is able to get database access, he will be only able to view the hashed output and not the actual password. Almost all computer passwords are stored in this fashion. When you create a password, the computer runs it through a hash function then stores only the result. Because the function is 'one-way', even if someone were to gain access to the file that stores that output, they shouldn't be able to figure out your password.
    When the computer prompts you to enter your password to log in, it will simply hash whatever you give it and then compare it to the stored hash of your password. This is often why passwords must be reset. Because the computer never stores your actual 'plain text' password and only a fingerprint of it, there is no way for it to tell you what your password was.
