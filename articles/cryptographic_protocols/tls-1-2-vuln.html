ROBOT Attack: What is it and how to avoid it?

A recent vulnerability was found with the way TLS uses RSA keys for encryption that would let an attacker decrypt any data that is sent over a TLS session. This defeats the purpose of confidentiality and should therefore be treated as a serious issue.

Several product manufacturers have provided patches to the problem here:
https://robotattack.org/?lipi=urn%3Ali%3Apage%3Ad_flagship3_pulse_read%3B0LalERDcQ8y2%2BHNBAvH4Rg%3D%3D#patches

NOTE: The above list will be updated as more patches are available.

In addition to patching servers, the recommendations from researchers now state that RSA encryption should be disabled from SSL/TLS cipher suites. This means all cipher suites that start with TLS_RSA should be disabled, for example:

TLS_RSA_WITH_AES_128_GCM_SHA256

Please note that the above is different from a cipher suite that uses both ECDHE and RSA. For example:

TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256

The above cipher suite is not vulnerable to the ROBOT attack. This is because of ECDHE keys being used as encryption keys and RSA being used for signature verification.

Action Taken to protect against a ROBOT attack.

In summary, administrators can perform the following to protect their servers:
  • Perform the ROBOT test on the server to discover if you have a vulnerability.
  • Contact your server vendor to determine if the server is vulnerable and if there are any patches.
  • If your server vendor does not have a patch, you may consider disabling TLS_RSA cipher suites.
  • Even if the ROBOT vulnerability is mitigated, you may consider disabling TLS_RSA cipher suites as they appear to be high risk and do not support perfect forward secrecy.
NOTE: NO action is required for SSL/TLS certificates as the certificate private key is not exposed by this vulnerability.

Logjam:

NOTE: This is not a specific vulnerability with SSLv3. This is vulnerability with a parameter that is configurable and continues to be used with all versions of SSL/TLS.

Logjam is a security vulnerability against a Diffie–Hellman key exchange ranging from 512-bit (US export-grade) to 1024-bit keys. It was discovered by a group of computer scientists and publicly reported on May 20, 2015. The Logjam attack allows a man-in-the-middle attacker to downgrade vulnerable TLS connections to 512-bit export-grade cryptography. This allows the attacker to read and modify any data passed over the connection. The attack is reminiscent of the FREAK attack, but is due to a flaw in the TLS protocol rather than an implementation vulnerability, and attacks a Diffie-Hellman key exchange rather than an RSA key exchange. The Logjam attack is a perfect example of an attack that was theoretical but became a very real threat to the current environment of readily available raw computing power.

Learn more about Logjam: https://blogs.cisco.com/security/understanding-logjam-and-future-proofing-your-infrastructure
Learn more about different key exchange option available in SSL/TLS: https://security.stackexchange.com/questions/20803/how-does-ssl-tls-work/20847#20847
